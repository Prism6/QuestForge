## 2. 개선 내용 상세

### 2-1. `utils/exceptions.py` 신규 — 커스텀 예외 계층

**문제:** `quest_generator.py`가 범용 `Exception`을 `raise Exception(f"...")` 형태로 재발생시켜
호출부에서 예외 종류를 구분할 수 없었다.

**해결:** 4단계 예외 계층 도입.

```
QuestForgeError (최상위)
├── QuestValidationError  — 필드 누락·값 범위 오류
├── QuestGenerationError  — API 호출·네트워크 오류
└── QuestParseError       — JSON 파싱 오류
```

**적용 위치:**

| 파일 | 변경 전 | 변경 후 |
|------|---------|---------|
| `quest_generator.py` | `raise Exception(f"퀘스트 생성 실패: ...")` | `raise QuestGenerationError(...) from e` |
| `quest_generator.py` | `except (ValueError, json.JSONDecodeError): raise` | `except QuestValidationError: raise` / `except json.JSONDecodeError: raise QuestParseError(...)` |
| `models.py` `from_dict` | `raise ValueError(f"필수 필드...")` | `raise QuestValidationError(...)` |
| `app.py` | `except Exception as e:` | `except QuestForgeError as e:` |

### 2-2. `utils/models.py` 개선 — `QuestType` Enum + `__post_init__` 검증

#### QuestType Enum

```python
class QuestType(str, Enum):
    MAIN       = "main"
    SUB        = "sub"
    DAILY      = "daily"
    REPEATABLE = "repeatable"
```

- `str` 상속: f-string·JSON 직렬화 시 문자열 값("main" 등)으로 동작 → 하위 호환 유지
- `QuestData.quest_type: str` → `QuestType` 으로 타입 강화
- `from_dict()` 에서 `QuestType(data["quest_type"])` 로 파싱 — 유효하지 않은 값은 `QuestValidationError`
- `to_dict()` 에서 `self.quest_type.value` 로 명시적 직렬화

#### `__post_init__` 유효성 검증

```python
def __post_init__(self) -> None:
    if not 1 <= self.difficulty <= 5:
        raise QuestValidationError(f"난이도는 1~5여야 합니다: {self.difficulty}")
```

`QuestData` 인스턴스가 생성되는 즉시 난이도 범위를 검증한다 (fail-fast 강화).

#### `utils/prompts.py` 연동

```python
QUEST_TYPE_KO_TO_EN: Dict[str, QuestType] = {
    "메인": QuestType.MAIN,
    ...
}
QUEST_TYPE_EN_TO_KO: Dict[QuestType, str] = {
    qt: ko for ko, qt in QUEST_TYPE_KO_TO_EN.items()
}
```

한국어·QuestType 간 변환 매핑도 Enum 기반으로 전환하였다.

### 2-3. DIP 완전 분리 — `AnthropicLLMClient` 모듈 분리

**문제:** `llm_client.py` (인터페이스 파일) 상단에 `from anthropic import Anthropic` 임포트가 존재했다.
이 때문에 `LLMClient` ABC를 임포트하는 것만으로 `anthropic` 라이브러리 의존성이 발생했다.
Mock LLMClient 를 사용하는 테스트에서도 `anthropic` 설치가 강제되었다 (DIP 위반).

**해결:** `AnthropicLLMClient` 를 전용 파일로 분리.

```
utils/llm_client.py      ← LLMClient ABC 만 존재. 외부 라이브러리 임포트 없음.
utils/anthropic_client.py ← AnthropicLLMClient 구현체. anthropic 라이브러리 사용.
```

| 파일 | 변경 전 | 변경 후 |
|------|---------|---------|
| `llm_client.py` | `LLMClient` + `AnthropicLLMClient` | `LLMClient` ABC 만 |
| `anthropic_client.py` | 없음 (신규) | `AnthropicLLMClient` 구현체 |
| `app.py` | `from utils.llm_client import AnthropicLLMClient` | `from utils.anthropic_client import AnthropicLLMClient` |

### 2-4. `utils/__init__.py` — 패키지 공개 API 정의

`utils` 패키지가 노출할 심볼을 `__all__` 에 명시하였다.
외부에서 내부 모듈 구조를 몰라도 `from utils import QuestData, QuestType` 등으로 직접 임포트 가능하다.

### 2-5. `tests/` — 유닛 테스트 30개 (전체 통과)

```
tests/
├── __init__.py
├── test_models.py           # QuestData 13개 테스트
├── test_quest_generator.py  # QuestGenerator 7개 테스트
└── test_export_strategy.py  # ExportStrategy 11개 테스트 (미포함 1개 제외)
```

#### 핵심 테스트 케이스

**`test_models.py`**
- 정상 데이터 → `QuestData` 반환
- `quest_type` 이 `QuestType` Enum 인스턴스인지 확인
- 4가지 quest_type 모두 파싱 가능
- 필수 필드 누락 → `QuestValidationError`
- 유효하지 않은 quest_type → `QuestValidationError`
- difficulty 범위 벗어남(0, 6) → `QuestValidationError`
- difficulty 경계값(1, 5) → 정상 통과
- `to_dict()` 후 `quest_type` 이 str "main" 으로 직렬화되는지 확인

**`test_quest_generator.py`**
- Mock LLMClient 로 실제 API 없이 `generate_quest()` 단위 테스트
- 코드 블록(` ```json ... ``` `) 응답 파싱 정상 동작
- 잘못된 JSON 응답 → `QuestParseError`
- API 오류 → `QuestGenerationError`
- 필드 누락 응답 → `QuestValidationError`
- `regenerate_quest()` 에서 genre/theme 보존 확인

**`test_export_strategy.py`**
- 단일 퀘스트 → JSON dict, 복수 → JSON list
- `quest_type` 이 str "sub" 로 직렬화 확인
- Excel 출력 bytes 반환 확인
- MIME 타입 및 파일 확장자 검증
